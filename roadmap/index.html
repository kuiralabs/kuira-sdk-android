    Roadmap¶

    This is the public wishlist for the Kuira Android SDK — patterns the SDK should bake in so the next dApp doesn't reinvent them.

    Items are added as real dApps (BBoard, Midnight Kicks, the Kuira Starter, CipherDefense) bump into the same friction. The list is deliberately bigger than the next release — most items wait until a second consumer hits the same gap. Each row records the workaround the example apps use today so a consumer hitting the same shape knows how to bridge it locally.

    Items are promoted out of this list when they ship.

    # Wish Why it'd help Today's workaround
    1 awaitIndexerSynced(blockHeight) primitive Replace fixed delay(5000) waits between deploy → first call and between sequential tx. Reduces total match time and makes flow deterministic. Hard-coded 3–8s sleeps in MatchManager (INDEXER_SETTLE_MS, POST_JOIN_SETTLE_MS).
    2 MidnightContract.stateFlow(): Flow<ContractState> StatePoller becomes a one-liner. Lets UI react to opponent actions without polling boilerplate. Caller writes a polling loop against the indexer GraphQL.
    3 Retry policy on contract.call(...) "Indexer says contract not found, wait + retry" is a near-universal pattern after deploy. Bake retry-with-backoff into the call surface. MatchManager.aiJoin loops 10× by hand, matching on "not found" substring.
    4 ✅ DONE — Serializable state snapshot for BlockStore backup Realized: MatchStore.snapshotBytes() / restoreFromBytes() round-trip the active-match set (address + role + secret key + nonces + picks) as a versioned blob, and MatchStoreBackupProvider (bound as the SDK's AppDataBackupProvider) packs it into SigilBackup.appMetadata so it rides into Block Store next to the seed. Reinstall → sigil restore repopulates unfinished matches. Completed games are pruned on Resolved (store.delete), so the blob stays lean. The long-term archive tier (history / stats / completed games beyond Block Store's size budget) is #30. (was: manual serialization in the app layer)
    5 First-class Player / Identity abstraction The two-player witness pattern (P1 secret key, P2 secret key, swap on each call) is a recurring shape. A Player with secretKey + coinPublicKey + witness-registration helper would shrink each call site. MatchManager passes raw ByteArray secrets through every helper.
    6 Auto force-resync of dust around contract calls Every tx after a deploy or another tx needs wallet.forceResyncDust() to see the new UTXO state. Should happen inside contract.call when needed. Manual forceResyncDust() between every tx in MatchManager.
    7 Deadline / timeout helper BigInteger.valueOf(System.currentTimeMillis() / 1000 + N) for unix-second deadlines is everywhere a circuit takes a deadline. Computed by hand in MatchManager.aiJoin.
    8 "Test-mode seed" path The shared test seed (TEST_SEED in KicksActivity) for fast iteration on faucet networks should be a single SDK opt-in, not a literal in every example app. Hex literal in KicksActivity.
    9a ✅ DONE — Lossless ledger reads via SDK (MidnightContract.ledger()) Promoted from wishlist to friction log + landed in 2026-05-20. The cell-hex parsing path turned out to be fundamentally lossy: MidnightConfig.queryState exposes a flattened JSON cell tree where Vector<N, Uint<8>> cells with zero elements lose positional info (each Uint<8>=0 serializes to a zero-length byte array, so the cell hex is a concatenation of only the non-zero elements). No dApp can recover the original arrays from this view. Fix: the SDK now offers MidnightContract.ledger() which spins up QuickJs, invokes the contract's own generated ledger(state) function, and marshals the typed return values back to Kotlin via MidnightLedger — the same lossless path the contract uses internally. Each typed accessor (getUint8, getUint64, getBoolean, getBytes, getVectorUint8, + OrNull variants) carries range/shape validation; missing fields throw MissingLedgerFieldException with the known-fields list, type mismatches throw WrongLedgerFieldTypeException — loud failures beat the silent-zeros the cell-hex parser produced. Kicks's ContractStateSnapshot and BBoard's BBoardRepository migrate to contract.ledger() and drop their bespoke parsers. Lossy cell-hex parsing in each dApp (ContractStateSnapshot.kt, BBoardRepository.kt), which silently misdecoded any Vector<N, Uint<8>> containing zero elements.
    9b Typed Ledger<T> codegen from .compact With #9a in place, the runtime decoding is lossless and shared. The remaining ergonomic gap is compile-time type safety: today consumers write ledger.getUint8("p1Score") instead of ledger.p1Score. A Gradle plugin (com.midnight.kuira.contract) that reads the compiled JS's ledger() descriptor and emits a typed Kotlin data class per contract (PenaltyLedger(val p1Shoots: IntArray, val p1Score: Int, ...)) would close it. Lower urgency now that the correctness problem is solved — defer until a second consumer (e.g. CipherDefense) needs it. String-keyed accessors with runtime field-name validation. Works correctly, just less ergonomic than a generated data class.
    10 Public block subscription MidnightSdk.indexerClient is private, so app code can't reach subscribeToBlocks() for push-based contract-state watching. Expose either the indexer or a thin MidnightContract.stateFlow() over it. StatePoller falls back to 3s polling.
    11 Kuira Gradle plugin for contract artifact sync Copying compiled .compact outputs into Android assets (penalty-contract.js + keys/*.{prover,verifier,bzkir}) is the same shape across every Kuira dApp, but each one hand-rolls a Gradle Copy task with hardcoded contract names and paths. A com.midnight.kuira.contract plugin should let dApps declare kuiraContract { source = "contract/src/managed/penalty" } (or auto-discover from a sibling module) and get sync-before-mergeAssets for free. app/build.gradle.kts registers a hand-rolled syncContractAssets Copy task with the contract path and circuit-key globs baked in.
    12 Typed witness factories for non-scalar types WitnessResult(null, ByteArray) works for Bytes<32> / Uint<8>, but for Vector<N, T> and other compound types the developer has to know the runtime's serialization (concatenated element bytes) and assemble it by hand. SDK should expose factory helpers like vectorWitness(length=5) { intArrayOf(L, C, R, L, R) } and boolWitness { true } that produce the right ByteArray for the underlying compact-runtime descriptor. MatchManager packs Vector witnesses by hand: byteArrayOf(*shoots.map(Int::toByte).toByteArray()).
    13 Compactc version pinning Compactc's --runtime-version must match the project's @midnight-ntwrk/compact-runtime npm version, but there's no in-repo way to declare which compactc version to use — developers find out via a runtime error after npm run compact writes incompatible bytecode. Either a .compact-version file in contract/ honored by the compact wrapper, or auto-pinning where compact compile reads package.json's pinned runtime version and switches itself. Kicks V3 requires compactc 0.30.0 (runtime 0.15.0); 0.31.0 is the user default and emits incompatible 0.16.0 bytecode. Workaround: invoke ~/.compact/versions/0.30.0/aarch64-darwin/compactc directly or run compact update 0.30.0.
    14 🔴 CRITICAL — Session auto-lock for the wallet SDK The decrypted seed lives in the one MidnightSdk instance now owned process-wide by MidnightSdkProvider (sdk:wallet-runtime, landed 2026-05-24). After one biometric on first SDK build, every subsequent value-bearing call (register, send, deploy, Kicks join/commit/reveal, payout claim) runs with zero re-authentication. The Keystore's 30s validity window is bypassed by SDK caching — we never re-call loadSeed, so Keystore is never re-consulted. A borrowed-while-foregrounded device drains funds in one tap. SDK needs three independent re-lock triggers: idle timeout (default 5–15 min), background timeout (re-lock on resume after N seconds backgrounded), and screen-lock observer (re-lock when device locks). Plus a manual "Lock now" affordance in the wallet panel. Now unblocked (2026-05-24): the provider owns the single cached SDK and exposes close(), so all three triggers + manual lock collapse to "call close(), force re-bootstrap (biometric) on the next call" in one place instead of per-consumer. The teardown seam now exists (MidnightSdkProvider.close()); no caller invokes it yet — no idle eviction, no foreground re-arm, no manual lock.
    15 🔴 CRITICAL — Sigil UX safety (no destructive-button-on-Error trap, no missing sign-out) The SigilStatusPanel has two compounding UX gaps that together make a sigil look deleted with one wrong tap. (a) When restoreSeed fails (e.g. user taps Restore on a Forged sigil but there's no cloud backup, or PRF fails, or any other BackupException), the catch block sets status to Error and ErrorBody renders a forge sigil button. forgeSigil → persistSigil unconditionally overwrites the existing prefs. One tap from "I just wanted to try restore" to "the sigil I had is gone." (b) There is no in-app sign out affordance from Forged state — the only way to leave a sigil is force-uninstall the app, so users in distress end up tapping any visible action including the forge-in-Error path. Required fixes: (1) signOut() on SigilPanelViewModel that wipes sigil_identity prefs + optionally deletes Block Store blob, status → None, biometric-gated; (2) sign-out button in ForgedBody under a "danger zone" divider with a confirmation modal; (3) restoreSeed's catch must distinguish "restore failed from None" (Error with forge button is fine — nothing to lose) from "restore failed from Forged" (return to Forged with a transient banner, never show forge); (4) ErrorBody's forge button only renders when previous state was None / BackupAvailable. Update (2026-05-24) — generalised + multi-wallet decision. The trap fires for any error while Forged, not just restore; the proximate harm is persistSigil overwriting the local sigil pointer (DID/cred/pubkey) with no guard. So beneath the UI fixes above, add a data-layer guard: persistSigil refuses to overwrite an existing sigil unless an explicit replace = true is passed — set only by the deliberate sign-out → forge path. Defense-in-depth, so a missed UI path can't silently destroy a sigil. Recoverability nuance: forging mints a new passkey and leaves the old one in GPM, but the panel offers no sign in with existing passkey (SigilSession.signIn exists but is unwired) — only forge (new) and restore-from-cloud. So an accidental forge orphans the old wallet unless a Block Store backup exists. Wire sign-in-with-existing-passkey as a recovery path. Multi-wallet (multi-sigil) considered + deferred (2026-05-24): the DID is PRF-derived 1:1 from a passkey, so "multi-sigil" = track + switch among multiple passkeys — the MidnightSdkProvider rebuilds the wallet per active sigil, plus per-sigil Block Store backup and a switcher UI. A v1.1+ product decision that departs from the one-sigil-one-identity vision; this item's lifecycle work (no silent overwrite + explicit sign-out/replace) is its prerequisite either way. Force-quit + restart the app to reload prefs (if they exist), or accept the visual deletion. No in-app safety.
    16 Resume-aware multi-step protocol orchestrator MatchManager is ~600 lines of state-machine + transitions + resume logic that's ~90% generic. Three bugs landed on 2026-05-20 — joinAsP2 precondition fail when state was already Deployed, awaitOpponentJoin precondition fail when state was already Joined, MatchReady.CONTINUE launching Unity's choice picker on a session where picks were already on chain — all reduce to "code path assumes I'm at step 1 but state is at step N." The fix in each case was the same shape: walk the steps and skip whichever ones are already reflected on chain. resumePlayAsP1 / resumePlayAsP2 are hand-rolled versions of this per role. SDK should provide a Protocol<S> DSL where each step declares (a) precondition on a typed MidnightLedger, (b) the action, (c) an "already done on ledger?" predicate. protocol.runFrom(contract.ledger()) would auto-skip steps the chain already shows landed and resume from the right point. Folds in #1 (settle waits become declarative timeouts on each step) and #6 (force-resync becomes a between-step concern). Per-dApp orchestrator + per-dApp resumePlayAsP* helpers + per-dApp precondition catches. Three live bugs in this session, all the same shape.
    17 MidnightContract.callIdempotent(circuit, args, isDoneOnLedger) Closely related to #16, but applicable outside a full multi-step protocol. Every action we wrote on 2026-05-20 had to add a catch-block for "the contract already reflects this transition" — joinAsP2 matches on the "WAITING phase" substring; awaitOpponentJoin had to learn to no-op when state was past Deployed. Each dApp re-invents the same try { call } catch { if (chain says it's done) ok } pattern, and missing it means a wasted tx + a Failed state on resume. SDK could ship contract.callIdempotent(circuit, args, isDoneOnLedger = { ledger -> ledger.getUint8("phase") > PHASE_WAITING }). The SDK reads the ledger first; if the predicate is true, returns synthetically without a chain call. Eliminates the per-dApp catch pattern that's now appeared in three places. Hand-rolled catch + substring-match pattern in joinAsP2, awaitOpponentJoin's new idempotent shortcut, and (implicitly) submitP1Picks/submitP2Picks whenever resume drives those.
    18 WitnessStore — encrypted, multi-slot witness persistence MatchStore is ~250 lines per dApp: EncryptedSharedPreferences setup, role-aware rehydration through rehydrateLocalIdentity, content-equality boilerplate on ByteArray fields (the same trap ContractStateSnapshot had — see #9a), Block Store cloud backup wiring through MatchStoreBackupProvider. The whole rehydrate-witnesses-after-process-death pattern is generic but every dApp implements it. Without this layer the resume orchestrator (#16) can't actually work — it needs the witnesses to drive the remaining steps. SDK should ship a WitnessStore(scope: String) keyed by (contractAddress, slotName), with PRF-encrypted persistence + Block Store backup baked in. dApp code becomes store.save(address, "p1Regulation", witnessBundle) / store.load(address, "p1Regulation"). Composes with #16: each Step declares the witness slot it owns; resume auto-rehydrates before walking. Per-dApp MatchStore + MatchStoreBackupProvider + manual rehydrateLocalIdentity for each role + content-equality boilerplate per witness type.
    19 MidnightContract.readOnly(...) factory We made coinPublicKey: ByteArray? nullable on 2026-05-20 to support polling contracts that don't have a wallet attached. But the read-only pattern still requires cargo: skip witness registration, omit cpk, remember not to call .call() until runtime. A future dApp author won't know that's a valid shape from reading the builder. A dedicated MidnightContract.readOnly(config) { contractJs = …; address = … } factory that returns a typed MidnightContractReadOnly interface with ledger() + stateFlow() but NO call/prepare/deploy would make the misuse compile-time impossible instead of surfacing as IllegalStateException at the first write call. Today's pattern: MidnightContract.create { ... } with implicit cpk-null sentinel + requireWriteCapable() runtime check inside every write method.
    20 MidnightContract.observeLedger(): Flow<MidnightLedger> Sibling of #10 — the SDK already has the infrastructure (private indexer subscription), what's missing is the dApp-facing shape. Today every dApp's StatePoller is a 3-second polling loop calling contract.ledger() + distinctUntilChanged. A push-based Flow<MidnightLedger> backed by subscribeToBlocks would (a) drop the per-dApp poller class entirely, (b) cut detection latency from ~3s to ≤ one block (~6s on PREPROD but ≤ 1 block deterministically), and (c) compose with #16: the protocol orchestrator subscribes to the flow and reacts to phase changes instead of awaiting fixed predicates with hand-tuned timeouts. StatePoller 3s loop in Kicks; once #10 lands without the typed-Flow shape, every dApp will still write that loop.
    21 compact-engine-testing artifact — FakeMidnightContract for state-machine unit tests While writing the 2026-05-20 precondition tests for resumePlayAsP1/resumePlayAsP2, the third bespoke test seam landed in TestableMatchManager: executeDeploy, executeJoinMatch, now readResumeSnapshot. Pattern is now obvious — every dApp will write the same test scaffolding. A published compact-engine-testing artifact shipping FakeMidnightContract (canned MidnightLedger snapshots, deferred-await knobs for "opponent commits now", invocation counters per circuit) would let dApp authors unit-test their state machines without spinning up QuickJs or the real chain. Folds into #16's Protocol DSL once it exists: the fake feeds canned ledger snapshots, the protocol walks the steps, the test asserts which steps fired. Three hand-rolled internal open suspend fun seams in MatchManager, plus the TestableMatchManager subclass per-dApp.
    22 🔴 CRITICAL — Open-SDK permissionless integration (drop default PasskeyConfig) core:identity:IdentityModule.providePasskeyConfig() hardcodes rpId = "nel349.github.io" + rpName = "Kuira" as the default Hilt binding. Any third-party dApp that consumes the SDK and forgets to override that provider silently lands on the maintainer's domain — and PRF then doesn't work for them unless the maintainer adds their package + cert SHA-256 to the maintainer-hosted assetlinks.json. That coupling means the SDK is effectively permissioned: every new dApp requires a PR to the maintainer's nel349.github.io repo. The post-PRF architecture is supposed to be permissionless (any dApp picks its own domain → hosts its own assetlinks.json → uses :sdk:wallet-seed and gets its own sigil-derived wallet with zero maintainer touch), but the default provider undermines that by giving wrong-but-working semantics to a misconfigured host. Required: (1) remove providePasskeyConfig from IdentityModule — host-provided with no fallback, so Hilt fails fast with a clear missing-binding error instead of silently routing through someone else's domain; (2) each of our example apps (BBoard, Kicks) ships its own ~5-line *IdentityModule providing its own PasskeyConfig (also gives per-app rpName branding in the passkey prompt — "Midnight Kicks" vs "Kuira"); (3) INTEGRATION.md at the SDK root showing third-parties the 4-step recipe (domain → assetlinks → Hilt module → maven deps). Cost: ~5 files (1 main repo, 1 per example, 1 doc). Removes the last "Kuira owns the authority" coupling and makes the SDK genuinely open. Default PasskeyConfig lives in IdentityModule pointing at nel349.github.io; new third-party dApp authors don't know they have to override it; if they don't, PRF fails until the maintainer manually adds them to a maintainer-hosted assetlinks.json.
    23 🔴 VITAL UX — Single-biometric sigil forge (PRF-on-create extension) Status (2026-05-24): the sign-in path is already single-biometric — SigilSession runs one multi-salt PRF assertion (eval.first=SIGIL_SALT + eval.second=SEED_SALT) covering both DID derivation and wallet seed. This item is now scoped to the forge/create path only. After the Path A2 sigil refactor, the first-run forge flow shows two consecutive biometric prompts: one for createPasskey (which registers the credential and returns its P-256 pubkey for KeyAuthorization), then a second for sigilIdentityProvider.deriveSigilDid (which does authenticateWithPrf(SIGIL_SALT) to derive the Ed25519 identity keypair). That's a regression from the pre-A2 single-prompt forge, and "tap fingerprint, wait, tap fingerprint again" is exactly the friction that drops first-run conversion. The fix is WebAuthn's PRF extension on create: the spec allows prf.eval.first = SIGIL_SALT to be passed into a registration ceremony, and supporting authenticators return the PRF output in the same response that delivers the pubkey + credentialId. One biometric, both outputs, identical security. Implementation surface: (1) extend PasskeyManager.createPasskey to accept prfSalt: ByteArray? = null and parse the PRF output out of the registration response's clientExtensionResults.prf.results.first field; add it as a nullable prfOutput on PasskeyRegistrationResult; (2) SigilPanelViewModel.forgeSigil passes SIGIL_SALT into the create call, then uses result.prfOutput?.let { Ed25519PrfSigilProvider.ed25519DidFromSeed(it) } to derive the DID directly when available — only falls back to the existing two-prompt path when prfOutput == null (older authenticators that don't support PRF-on-create); (3) update the existing Ed25519PrfSigilProvider.ed25519DidFromSeed test plus add an end-to-end test that pins the "create returns PRF → no second prompt" path. Coverage note: Google Password Manager added PRF-on-create support in Chrome 132 / GMS Core ~24.x (2024); hardware authenticators (YubiKey 5C NFC etc.) have it; the fallback handles the rest. Cost: ~3 files, mostly mechanical JSON parsing for the assertion-extension field. Two biometric prompts on first forge — the user has to authenticate twice in immediate succession with no semantic distinction between the two prompts. Subsequent sign-ins are unaffected (already one prompt).
    24 🔴 CRITICAL — Sovereign recovery-phrase export (reveal the PRF-derived BIP-39 mnemonic) Today a Sigil-onboarded wallet's only recovery path is re-deriving the seed from the passkey, which rides on Google/Apple account sync. If that account is lost, locked, or unavailable in-jurisdiction (Google services blocked), the user has no way out — the funds are gone; "self-custody" is really platform-custody of the recovery path. The architecture can already close this: the PRF output is valid 32-byte BIP-39 entropy, and SeedDeriver already transiently materializes it as a 24-word mnemonic (its own "known limitation" comment) via BIP39.entropyToMnemonic. Promote that incidental transient into an intentional, biometric-gated revealRecoveryPhrase(activity) SDK primitive that returns a wipeable holder and zeroes entropy/seed in finally, then wire the existing RecoveryPhraseWireframe into production Settings nav. Result: the user can write down 24 words and import them into any BIP-39 wallet — a sovereign exit independent of Google/Apple. This is the load-bearing piece for the self-custody / web2.5 / international story; without it the promise is false for anyone who loses platform access. Must read the live funding seed (PRF-derived for Sigil wallets, random for create-onboarded ones) so the exported words match the seed actually holding funds, and must be FLAG_SECURE + no-default-clipboard + never-logged. Half-built round trip: import-from-phrase works (onboarding restore validates + derives via BIP39.mnemonicToSeed), but export does not — RecoveryPhraseWireframe lives in the dev portal only and Settings' "View recovery phrase" is a no-op. No production way to see the words the Sigil derives.
    27 🟡 Delegated access keys — scoped, time-bounded authorization for non-sibling keys The KeyAuthorization payload (Magic + rootPubkey + accessPubkey + scope + timestamp + expiry, ECDSA-signed by the passkey) was originally shipped as a BBoard demonstrator. We removed the demonstrator in the post-A2 cleanup because under PRF-derived sigil + wallet, both keys are siblings of the same passkey — they share a common root via PRF salt domain separation, so a separate authorize attestation is redundant for the local wallet use case. The type stays in core:identity (KeyAuthorization.kt, AuthorizationStore.kt, AuthorizationRecord.kt, plus IdentityModule's Hilt provider for the store), waiting for the real consumer it was designed for: signing an attestation that authorizes a SEPARATE, non-sibling key to act on the sigil's behalf. Use cases: (a) a remote agent / browser session signing transactions on the user's behalf for 24 hours with AuthorizationScope.SEND_ONLY; (b) a third-party dApp the user wants to grant transient permissions to without sharing the passkey; (c) a future multi-sig where one device's sigil authorizes another device's access key. None of those exist yet, so no UI consumer exists; the next iteration's design work is (1) UX for "review and approve this delegation request" coming from an external scope, (2) revocation mechanism (write a new authorization with expiry in the past, or maintain a revocation list), (3) on-chain or off-chain verification — verifiers need to know which Ed25519 sigil DID signed this P-256 authorization, which is awkward because the DID encodes Ed25519 but the signature is over P-256. Solution direction: sign authorizations with the Ed25519 sigil key (derivable from PRF on demand) rather than the passkey P-256 — self-contained verification from the DID alone. The demonstrator was the only consumer; removing it stranded the type. No active flow uses authorization today. Future delegated-key consumers will need to start from scratch UI-wise, but the cryptographic primitives + persistence are ready.
    28 🟡 Choose which identity/account backs a sigil at forge time The sigil is PRF-derived from a Google Password Manager passkey, and WebAuthn deliberately hides the account from the relying-party app — it can't choose, or even see, which account answers. GPM serves its one configured account. So a user with multiple Google accounts signed into a device has no way, inside our app, to bind a sigil to a chosen account: two devices (or two users) whose GPM points at the same account collide to the same sigil. This bit hard during PvP testing — two emulators resolved to one wallet, with no in-app lever to separate them. Two directions, both worth having: (i) a debug-only per-device identity seam so two installs (even on the same account) get distinct sigils for local/PvP testing without Google-account juggling; (ii) product-level identity selection — since the platform won't expose account choice, the app must own picking among multiple identities (this is the multi-sigil direction, extends #15). The constraint to record: you cannot solve this with an in-app "pick this Google account" button over the passkey API — that choice lives in Google Password Manager / the system sheet, outside the app. Switch Google Password Manager's account in Android Settings before forging (system-level, no guarantee of an in-sheet account picker), or use entirely separate Google accounts per device. No in-app choice.
    29 🟡 Rematch re-pairing — share / reuse / link the new contract REMATCH currently auto-deploys a fresh contract (P1) or routes to the join screen (P2), but there's no friendly bridge between the two: P1 has a new address and no smooth way to push it to the same opponent, and P2 lands on a blank join screen with no idea of the new address. The only link between players is the contract, and a resolved one is single-use, so "play the same opponent again" falls back to re-doing QR/deep-link matchmaking by hand. Three directions worth weighing as game logic, not just UX: (i) reusable/extensible contract — make penalty.compact replayable (a resetMatch circuit that re-opens commit phase for the same two registered players, or an N-matches-per-contract series with a running aggregate score), so REMATCH is a single circuit call with zero re-pairing; (ii) linked contracts — the new contract references the previous (prevMatch address) so each side can fetch the opponent's identity/stake from the old one and auto-join without re-sharing; (iii) a lightweight re-pair channel — P1's REMATCH deploys + the new address rides back to P2 over the existing relay / a deep-link the app surfaces as a one-tap "your opponent wants a rematch — join?" prompt. (i) is the cleanest game-feel but needs contract work + a stake-handling rethink (who funds the rematch pot); (ii)/(iii) keep the one-match-one-contract model but add a pairing hop. REMATCH → P1 re-runs CREATE (manual QR/link share), P2 → blank Join screen (manual address entry). No auto-pairing, no contract reuse, no link to the prior match.
    30 🟡 Two-tier persistence — Block Store for active recovery, Drive appDataFolder for the long-term archive Block Store is a small, transparent device-transfer vault (a few KB) — right for the critical recovery set (seed + active-match witnesses, already shipped via #4), wrong for archival data. As concurrent matches grow, the seed + per-match witnesses (each ~32B key + 32B×N nonces + picks) share that tiny budget; and completed-game history / stats / saved replays can't live there at all. Two complementary directions: (a) Tier-2 archive on Google Drive appDataFolder — a hidden, per-account, quota-bounded (effectively unlimited for metadata) folder for completed-match history, head-to-head stats, and replay payloads; sigil-derived-key encrypted before upload (same "we encrypt, cloud is just transport" stance as the Block Store blob), fetched lazily for a "match history" screen. (b) Shrink Tier-1 to near-zero by deriving each match's witnesses (secret key + commit nonces) deterministically from the sigil seed + contract address (+ round) over an HD path, instead of backing the random bytes up — then Block Store only needs the list of active contract addresses (tiny), or nothing if the indexer can enumerate "contracts where my key participates" (a participant-indexed query the indexer doesn't expose yet — the real discovery gap for "fetch my active matches after restore" without a local list). (a) is the user-facing "history that survives a device hop"; (b) is the scalability fix so Tier-1 never outgrows Block Store. Completed-game archive explicitly deferred for now. Active matches round-trip through Block Store (#4); no archive tier (completed games are deleted on resolve), and Tier-1 stores full random witnesses per match rather than deriving them, so it scales only to a handful of concurrent matches before bumping the Block Store ceiling.
    31 🟡 Silent app-state backup + lifecycle auto-trigger — make active-match cloud recovery set-and-forget Today the cloud backup (the only thing that makes a match survive uninstall) is manual: the sigil panel's Backup button → AppStateBackup.backup(activity) → authenticateWithPrf(BACKUP_SALT), i.e. a biometric prompt every time. So you can't auto-back-up when the app backgrounds/dies (no prompt possible there), and a match created-then-uninstalled before a manual Backup is lost. Key realization: post-PRF the backup blob carries only app metadata (match state) — the seed/sigil are PRF-reconstructed on any device, not in the blob. So we're stashing game state, not funds, and the per-backup biometric is overkill. SDK change: offer a no-prompt app-state backup that derives the appMetadata AES key from the already-cached session (unlocked seed) instead of a fresh PRF assertion — runs only while the session is unlocked (interacts with #14's auto-lock; back up before locking) and must run in the live process (a deferred WorkManager job is a fresh process with no cached key). App side (~20 lines): on ProcessLifecycleOwner ON_STOP, if the active-match snapshot diffs from the last backup (hash compare), fire the silent backup; best-effort — if the process is reaped mid-upload the diff still reads "changed" so the next background retries, and the local EncryptedSharedPreferences store always covers same-device resume. Note: ON_STOP is the reliable hook — onDestroy/process-kill give no guaranteed callback for a network upload. Cloud backup is a manual sigil-panel tap (biometric-gated); no automatic capture on leave, so a match made-then-uninstalled before a manual Backup never reaches Block Store.
    32 🟡 Upstream the DustLocalState Merkle-root accessors to midnightntwrk/midnight-ledger kuira-crypto-ffi calls state.commitment_root() and state.generation_root() in 18 places across balance_ffi.rs and dust_ffi.rs (not just diagnostics — balance_ffi.rs uses them in the live balance path).     33 Contract proving-key DX — ensureBlsParams(k), installCircuitKeysFromAssets, ensureContractKeys Every Compact dApp that ships proving keys in APK assets hits the same three gaps. (a) The wallet bundle only ships bls_midnight_2p13/14/15, so small contracts that need a smaller param set (counter needed 2p5, only 6.5KB) fail with BLS params file not found on cold install — no public way to fetch arbitrary BLS sizes despite S3_BASE_URL already being a constant in ProvingKeyManager. (b) The asset → cacheDir → installCircuitKeysForProving staging is ~30 lines of boilerplate every dApp re-implements (open three asset streams, stage to cacheDir, hand to the installer). (c) There's no single "make sure everything proves on the first call" entry point that composes (a) + (b) the way ensureWalletKeysAvailable already composes the wallet path. Three additions to ProvingKeyManager: (1) ensureBlsParams(k: Int) downloads bls_midnight_2p{k} from S3_BASE_URL on demand; (2) installCircuitKeysFromAssets(context, contractName, circuitNames) does the cacheDir staging in-SDK so dApps drop their copy block; (3) ensureContractKeys(...) composes the two with ensureWalletKeysAvailable, mirroring the existing wallet shape. Surfaced during the 2026-05-30 starter test-of-fire: increment failed with BLS params file not found: bls_midnight_2p5 until the starter manually bundled 2p5 in assets and wrote the staging logic. Folds in the BBoard/Kicks dev-only pattern: both currently rely on adb push /data/local/tmp/bboard_keys/* for their proving keys, which is impossible for any non-dev install — once #33 lands they can both move to APK assets without rewriting the prover path. Starter ships bls_midnight_2p5 in app/src/main/assets/managed/counter/bls/ and CounterContract.installProvingKeys writes ~30 lines of cacheDir staging by hand. BBoard + Kicks still require adb push /data/local/tmp/bboard_keys/* before each first-call, which isn't a production pattern.
    34 Benchmark on-device proving latency (real devices, per circuit complexity) The "proofs run on-device, offline" claim is architecturally verified, but there's no defensible latency number to publish. Observed-but-unpublishable figures so far are all arm64-emulator + small circuits + only the Dust-spend leg (balance_ffi: Dust tx proved in 0.5–0.65s; contract circuit proof itself 0.00–0.10s for counter/BBoard). A public number needs: (1) a range of physical devices (low/mid/high tier), (2) circuit-complexity tiers (trivial counter → Kicks' 10-round penalty → heavier), (3) the Dust leg vs circuit leg broken out separately (they're different work), (4) a reproducible harness so the number is re-measurable, not a one-off log line. Until then the capabilities doc states only the architectural claim (on-device, offline, no proof server) with no time figure. Rule going in: never publish a significant performance number without a reproducible measurement behind it. Capabilities/showcase copy omits proving latency entirely; only emulator log lines exist, which must not be quoted as guarantees.
