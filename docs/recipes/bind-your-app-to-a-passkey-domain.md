# Bind your app to a passkey domain

**Outcome:** your APK is signed with a stable keystore, your domain
serves an `assetlinks.json` that lists that keystore's fingerprint
against your app's package name, and the passkey API will honour the
`rpId` you pass to `PasskeyConfig` — no `RP_ID_MISMATCH`, no `PRF
authentication failed`, no silent prompt dismissal.

<div data-copy-prompt="https://raw.githubusercontent.com/kuiralabs/kuira-sdk-android/main/docs/recipes/bind-your-app-to-a-passkey-domain.md"
     data-task="Configure passkey domain binding for this Android app — generate or locate the signing keystore(s), compose an assetlinks.json with the right fingerprints, host it under .well-known/, and verify it. Cover both debug and release builds."></div>

## Mental model

Three things have to line up:

1. **A keystore** signs the APK. The signature embeds the keystore's
   SHA-256 cert fingerprint in every APK built from that keystore.
2. **An `assetlinks.json`** at `https://<rpId>/.well-known/assetlinks.json`
   declares "this package, signed with this fingerprint, is allowed to
   speak for this domain."
3. **Your code** passes `rpId = "<that-same-domain>"` to `PasskeyConfig`.

At passkey-create time the Android `CredentialManager` resolves your
package's signing fingerprint, fetches your `rpId`'s `assetlinks.json`,
and refuses the ceremony unless the fingerprint shows up in the file.
That's the only check — but it's all-or-nothing.

Debug and release builds have **different** signing fingerprints by
default, so production apps need their release fingerprint added to the
same `assetlinks.json`.

## Part 1 — Find your fingerprints

### Recommended: `./gradlew signingReport`

The fastest, AGP-native way. Prints the SHA-256 for every
build-variant × flavor combination at once:

```bash
./gradlew :app:signingReport
```

Output (truncated):

```
Variant: debug
Config: debug
Store: /Users/you/.android/debug.keystore
Alias: AndroidDebugKey
SHA-256: AB:CD:EF:12:34:…
Valid until: 2052

Variant: release
Config: release
Store: /path/to/release.keystore
Alias: release-key
SHA-256: 11:22:33:44:55:…
Valid until: 2049
```

The `SHA-256:` line is what goes into `assetlinks.json`. Strip the
colons or keep them — both forms are accepted; we'll keep them for
readability.

### Fallback: raw `keytool`

If you're on a machine without the project (e.g. inspecting a release
keystore on a CI runner), the raw command:

```bash
keytool -list -v \
  -keystore <path-to-keystore> \
  -alias <alias> \
  -storepass <store-password> \
  | grep "SHA256:"
```

For Android Studio's auto-created debug keystore:

```bash
keytool -list -v \
  -keystore ~/.android/debug.keystore \
  -alias androiddebugkey \
  -storepass android -keypass android \
  | grep "SHA256:"
```

The debug keystore is auto-created the first time you build any
Android project on this machine — if `~/.android/debug.keystore`
doesn't exist, run `./gradlew :app:assembleDebug` once and it will
appear.

## Part 2 — Configure release signing

Debug works for development. The moment you publish to the Play Store,
distribute to testers, or run a release build for benchmarking, you
need a **release keystore** — generated by you, stored securely, and
backed up.

!!! danger "If you lose this keystore, you cannot update your app"
    Google Play binds your published app to the keystore that first
    signed it. A lost release keystore means the next version can't
    ship under the same listing. Store backups in two places.

### Generate the keystore

```bash
keytool -genkey -v \
  -keystore release.keystore \
  -alias release-key \
  -keyalg RSA -keysize 2048 \
  -validity 10000
```

Answer the prompts (name, org, locality, two letters for country). You
end up with `release.keystore` in the current directory — move it
somewhere outside your repo.

### Wire it into Gradle without committing secrets

Create `keystore.properties` at the project root — **not committed**,
add to `.gitignore`:

```properties title="keystore.properties (gitignored)"
releaseStoreFile=/Users/you/keystores/release.keystore
releaseStorePassword=…
releaseKeyAlias=release-key
releaseKeyPassword=…
```

Read it from `app/build.gradle.kts`:

```kotlin title="app/build.gradle.kts"
import java.io.FileInputStream
import java.util.Properties

val keystoreProps = Properties().apply {
    val f = rootProject.file("keystore.properties")
    if (f.exists()) load(FileInputStream(f))
}

android {
    signingConfigs {
        if (keystoreProps.containsKey("releaseStoreFile")) {
            create("release") {
                storeFile = file(keystoreProps["releaseStoreFile"] as String)
                storePassword = keystoreProps["releaseStorePassword"] as String
                keyAlias = keystoreProps["releaseKeyAlias"] as String
                keyPassword = keystoreProps["releaseKeyPassword"] as String
            }
        }
    }
    buildTypes {
        release {
            signingConfigs.findByName("release")?.let { signingConfig = it }
            // …
        }
    }
}
```

The `if`-guards mean the build still works on machines where
`keystore.properties` isn't present (e.g. someone running
`assembleDebug` from a fresh clone). Release builds on those machines
will be unsigned, which is the correct behaviour.

### CI signing via GitHub Actions

For CI to produce a signed release build, you need the keystore bytes
and credentials available to the runner without checking them in.
Store the keystore as a base64-encoded secret:

```bash
base64 -i release.keystore | pbcopy   # macOS
# paste into GitHub repo → Settings → Secrets → KEYSTORE_BASE64
```

Add four secrets: `KEYSTORE_BASE64`, `KEYSTORE_STORE_PASSWORD`,
`KEYSTORE_KEY_ALIAS`, `KEYSTORE_KEY_PASSWORD`.

In the workflow:

{% raw %}
```yaml title=".github/workflows/release.yml"
- name: Reconstruct keystore
  run: |
    echo "${{ secrets.KEYSTORE_BASE64 }}" | base64 -d > "$RUNNER_TEMP/release.keystore"
    cat <<EOF > keystore.properties
    releaseStoreFile=$RUNNER_TEMP/release.keystore
    releaseStorePassword=${{ secrets.KEYSTORE_STORE_PASSWORD }}
    releaseKeyAlias=${{ secrets.KEYSTORE_KEY_ALIAS }}
    releaseKeyPassword=${{ secrets.KEYSTORE_KEY_PASSWORD }}
    EOF

- name: Assemble release
  run: ./gradlew :app:assembleRelease
```
{% endraw %}

## Part 3 — Compose your `assetlinks.json`

The schema is a JSON array — one entry per (relation, target-app) pair.
For passkeys you want both `handle_all_urls` (App Links) and
`get_login_creds` (passkey-specific) relations.

### Single fingerprint (debug only)

```json title="assetlinks.json"
[
  {
    "relation": [
      "delegate_permission/common.handle_all_urls",
      "delegate_permission/common.get_login_creds"
    ],
    "target": {
      "namespace": "android_app",
      "package_name": "com.example.myapp",
      "sha256_cert_fingerprints": [
        "AB:CD:EF:12:34:56:…"
      ]
    }
  }
]
```

### Debug + release fingerprints (recommended)

You can list **multiple fingerprints in the same target** — perfect for
covering both your debug and release signing certs without hosting two
files:

```json title="assetlinks.json" hl_lines="11 12"
[
  {
    "relation": [
      "delegate_permission/common.handle_all_urls",
      "delegate_permission/common.get_login_creds"
    ],
    "target": {
      "namespace": "android_app",
      "package_name": "com.example.myapp",
      "sha256_cert_fingerprints": [
        "AB:CD:EF:12:34:56:…",
        "11:22:33:44:55:66:…"
      ]
    }
  }
]
```

The same domain now honours passkeys from both your debug build (you,
testing locally) and your release build (users on the Play Store).

### Multiple apps on the same domain

If you ship more than one app to the same `rpId`, add one entry per
package — each with its own fingerprint list:

```json title="assetlinks.json"
[
  {
    "relation": ["delegate_permission/common.get_login_creds"],
    "target": {
      "namespace": "android_app",
      "package_name": "com.example.appone",
      "sha256_cert_fingerprints": ["AB:CD:…"]
    }
  },
  {
    "relation": ["delegate_permission/common.get_login_creds"],
    "target": {
      "namespace": "android_app",
      "package_name": "com.example.apptwo",
      "sha256_cert_fingerprints": ["12:34:…"]
    }
  }
]
```

## Part 4 — Host it

The file MUST be reachable at exactly
`https://<rpId>/.well-known/assetlinks.json`, served as `Content-Type:
application/json`, over HTTPS, with no redirect, no auth wall, and no
trailing-slash funkiness.

### GitHub Pages

Put `assetlinks.json` at `<repo>/.well-known/assetlinks.json` on the
branch GitHub Pages serves from (`main` or `gh-pages`). GitHub Pages
serves `.json` as `application/json` by default — no extra config.

```
your-pages-repo/
  .well-known/
    assetlinks.json
```

Verify:

```bash
curl -I https://<your-handle>.github.io/.well-known/assetlinks.json
# HTTP/2 200
# content-type: application/json
```

If you use a custom domain on GitHub Pages, your `rpId` is that custom
domain, and you set the `CNAME` file as usual.

### Vercel

Put it at `public/.well-known/assetlinks.json` so it ships as a static
asset:

```
your-vercel-project/
  public/
    .well-known/
      assetlinks.json
```

Add a `vercel.json` header rule to force the right content type:

```json title="vercel.json"
{
  "headers": [
    {
      "source": "/.well-known/assetlinks.json",
      "headers": [
        { "key": "Content-Type", "value": "application/json" }
      ]
    }
  ]
}
```

### Cloudflare Pages

Same shape — put it under your build output's static directory
(`public/.well-known/assetlinks.json` for most frameworks). Cloudflare
serves `.json` as `application/json` by default. Cache-control is
opaque — purge the cache after every edit:

```bash
# Cloudflare dashboard → Caching → Purge Cache → Custom Purge
# URL: https://<rpId>/.well-known/assetlinks.json
```

### Custom server

Any HTTP server that can:

- Serve a file at `/.well-known/assetlinks.json`
- Send `Content-Type: application/json`
- Return `200` (not 301/302) on `GET` and `HEAD`
- Reach the file over public HTTPS with no auth

…will work. Nginx snippet:

```nginx title="nginx.conf"
location = /.well-known/assetlinks.json {
    default_type application/json;
    add_header Cache-Control "public, max-age=300";
}
```

## Part 5 — Verify

### `curl -I` smoke test

```bash
curl -I https://yourapp.example.com/.well-known/assetlinks.json
```

Acceptable response:

```
HTTP/2 200
content-type: application/json
```

Common red flags:

- `HTTP/2 301` / `HTTP/2 302` — there's a redirect (often http→https,
  www→apex, or trailing-slash). The passkey API does not follow
  redirects. Fix the redirect so the file is served at the canonical URL.
- `HTTP/2 403` / `HTTP/2 401` — auth wall. Bypass it for `/.well-known/*`.
- `content-type: text/html` or `text/plain` — your server's
  not honouring the `.json` extension. Add the explicit header (see
  Vercel / Nginx snippets above).
- `HTTP/2 404` — file not deployed; or your `rpId` doesn't match the
  hostname you `curl`ed.

### Google's Statement List Tester

A second pair of eyes — verifies the JSON shape, fingerprint
formatting, and the relations Google's tooling expects:

[Google Digital Asset Links Tester →](https://developers.google.com/digital-asset-links/tools/generator)

Paste your hosting URL + package name + fingerprint; it reports any
mismatch with a precise error string.

### On-device verification

Build the debug APK, install, tap Forge:

```bash
./gradlew :app:installDebug
adb shell am start -n com.example.myapp/.MainActivity
# (then tap Forge in the panel)
```

If everything's right, you see the biometric prompt → success → sigil
panel transitions to `Forged`. If it fails, see Troubleshooting below.

## Troubleshooting

| Symptom (in app, logcat, or biometric prompt) | Likely cause | Fix |
|---|---|---|
| `RP_ID_MISMATCH` | `assetlinks.json` unreachable, wrong content-type, or fingerprint mismatch | `curl -I` checks above; verify the fingerprint matches `./gradlew signingReport` for the variant you installed |
| `PRF authentication failed` | Same as above — `assetlinks.json` doesn't validate, so the passkey API never reaches the PRF step | Same checklist |
| Biometric prompt appears, then dismisses silently | Same as above, surfaced as a UX no-op instead of an exception on some devices | Same checklist |
| Forge succeeds on debug, fails after release build | You added the debug fingerprint to `assetlinks.json` but not the release fingerprint | Add the release SHA-256 to `sha256_cert_fingerprints` in the same target |
| Forge worked yesterday, breaks today | Either you re-generated the debug keystore (wipes the fingerprint), or your hosting platform changed its cache/redirect behaviour | Re-run `signingReport` and re-publish `assetlinks.json` if the fingerprint changed; purge CDN if not |
| Worked before, broken after `adb install -r` | `adb install -r` does NOT refresh credential-manager state on some devices | `adb uninstall com.example.myapp && ./gradlew :app:installDebug` |
| Works on emulator, fails on physical device | Different Google Play Services account; passkeys are per-account | Sign into the same GPM account on both, or test the restore flow deliberately |
